Research Prototype Documentation

SafeMAP translates only what can be accepted as fully safe Rust.

SafeMAP is a safe-first C-to-Rust migration prototype. It analyzes C code, classifies what is statically eligible for safe migration, plans Rust-native APIs, generates or rewrites Rust, validates the result, and reports what was accepted or rejected.

What SafeMAP Is

SafeMAP stands for Safe Migration through Analysis-Guided Planning. It is built for research, not production deployment. Its job is to show that a subset of C can be migrated to idiomatic, fully safe Rust when the tool is strict about eligibility, APIs, validation, and failure reporting.

SafeMAP does

  • analyze C functions and pointer usage;
  • detect common migration idioms;
  • classify unsupported code honestly;
  • generate migration plans as research artifacts;
  • accept Rust only under strict safety rules;
  • compare against C2Rust baseline output.

SafeMAP does not

  • translate arbitrary C projects completely;
  • treat unsafe reduction as full success;
  • hide unsupported constructs;
  • claim formal semantic equivalence;
  • replace manual review for complex systems code.

Research Claim

The current MVP supports this conservative claim:

SafeMAP is a safe-first C-to-Rust migration prototype that generates fully safe Rust for a restricted set of statically eligible C functions. Unsupported units are rejected or reported, final Rust is evaluated under #![forbid(unsafe_code)], and results are compared against a C2Rust baseline using explicit safety and validation metrics.

What “fully safe accepted” means

  1. The generated crate or module compiles with #![forbid(unsafe_code)].
  2. The accepted Rust contains no unsafe, unsafe fn, or unsafe impl.
  3. The accepted public API exposes no raw pointers such as *const T or *mut T.
  4. Available validation passes.
  5. Repair steps do not introduce unsafe Rust.

Architecture

SafeMAP has a safe-first main lane and a C2Rust baseline lane. The lanes are intentionally separate so baseline output cannot be mistaken for a SafeMAP success.

Pipeline Stages

1

Ingestion

Discovers C files, headers, build files, tests, entrypoints, and compile database information.

2

C Analysis

Extracts functions, signatures, calls, pointer facts, structs, globals, and idioms.

3

Eligibility

Classifies each translation unit before attempting safe migration.

4

Planning

Writes JSON migration plans with target signatures, idioms, constraints, and validation rules.

5

Synthesis or Rewrite

Uses deterministic MVP synthesis or an LLM client to generate safe Rust candidates.

6

Validation and Reporting

Runs Cargo tools, collects metrics, and writes research reports.

Eligibility Categories

Eligibility classification prevents unsupported C from being counted as successful safe migration.

Category Meaning Counts as eligible?
safe_translatable Can be translated without major API change. Yes
safe_translatable_with_api_change Can be translated safely with Rust-native API changes. Yes
requires_safe_wrapper May need unsafe internals hidden behind a safe wrapper. No for strict SafeMAP acceptance
requires_manual_refactor Needs restructuring before safe Rust can be generated. No
unsafe_required Operation inherently requires unsafe Rust or FFI. No
unsupported Current prototype does not handle the construct. No

Supported C Idioms

Pointer-length arrays

int *arr, int len becomes &[i32] or &mut [i32].

Output parameters

void get(int *out) becomes a returned value or tuple.

Error code plus output

int parse(..., int *out) becomes Result<i32, i32>.

Nullable pointers

if (p == NULL) patterns become Option<&T>.

Manual allocation

Simple ownership patterns are planned toward Vec<T> or Box<T>.

Booleans and constants

Simple integer idioms can be represented using Rust-native types.

Use Python 3.11 or newer.

python -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -e ".[dev,rust-analysis]"

Verify

python -m safemap.cli --help
pytest

Optional External Tools

  • cargo, rustc, and clippy for Rust validation;
  • clang and libclang for C analysis;
  • c2rust for baseline generation;
  • bear for Make-based compile database capture;
  • Miri for optional undefined-behavior checks.

Running SafeMAP

Full run

python -m safemap.cli run \
  --input examples/output_parameter \
  --output results \
  --config safemap.example.yaml

Stage-by-stage run

python -m safemap.cli ingest --input examples/output_parameter --output work
python -m safemap.cli analyze-c --workdir work
python -m safemap.cli translate-baseline --workdir work
python -m safemap.cli analyze-rust --workdir work
python -m safemap.cli plan --workdir work
python -m safemap.cli rewrite --workdir work --config safemap.example.yaml
python -m safemap.cli repair --workdir work --config safemap.example.yaml
python -m safemap.cli validate --workdir work --config safemap.example.yaml
python -m safemap.cli report --workdir work

Metrics

SafeMAP reports success through strict safe acceptance, not just reduction in unsafe code.

Primary fully_safe_translation_unit_acceptance_rate
Safety unsafe blocks, unsafe functions, raw pointers, public raw-pointer APIs
Validation compile, tests, clippy, Miri, differential testing
Failure unsupported, unsafe required, manual refactor, repair failed, validation failed

C2Rust Baseline

C2Rust is valuable as a baseline and reference artifact. It often emits unsafe Rust and C-style APIs, which is exactly why SafeMAP reports it separately.

Important: C2Rust output is not counted as a SafeMAP success unless it independently compiles under #![forbid(unsafe_code)] and passes the same acceptance rules.

Common local issue

C2Rust 0.22 can require older LLVM runtime libraries. If the binary cannot find libclang-cpp.so.14, set the local LLVM path variables shown in the configuration section.

LLM Rewrite and Repair

The LLM stage is optional and provider-abstracted. SafeMAP currently uses an OpenAI-compatible client, so Gemini can be used through Gemini's OpenAI-compatible endpoint.

Prompt constraints

  • No unsafe.
  • No raw pointer public API.
  • No extern "C" in safe translated logic.
  • No placeholders such as todo!() or unimplemented!().
  • Preserve the target signature from the migration plan.

Responses that introduce forbidden unsafe constructs are rejected before being patched into source files.

Research Paper Workflow

  1. Freeze the benchmark set.
  2. Run all benchmark modes and keep the run artifacts.
  3. Use reports/metrics.json and reports/comparison.csv for tables.
  4. Describe unsupported cases as results, not as tool crashes.
  5. Report C2Rust baseline separately from SafeMAP final output.
  6. Include threats to validity around benchmark size, LLM nondeterminism, and partial C support.

Do not claim

  • full automatic C-to-Rust migration;
  • formal equivalence;
  • safe migration of arbitrary C;
  • success based only on fewer unsafe blocks.

Limitations

SafeMAP is intentionally conservative. Rejection is a valid outcome when static evidence is insufficient.

  • Fallback C parsing is limited compared with libclang.
  • Complex ownership and aliasing are not fully modeled.
  • Current deterministic synthesis covers only a small MVP set.
  • Differential testing is not always applicable.
  • LLM output may vary by model and prompt.
  • C2Rust depends on local LLVM and Rust compiler compatibility.

Troubleshooting

--config: expected one argument

The shell split the command after --config. Use a backslash for multiline commands. 

python -m safemap.cli run \
  --input examples/output_parameter \
  --output results \
  --config safemap.example.yaml

Missing LLM API key

Set the environment variable named by llm.api_key_env.

C2Rust cannot find libclang-cpp.so.14

Set SAFEMAP_C2RUST_LIB_DIR or install the LLVM 14 runtime libraries.

C2Rust baseline fails to compile on stable Rust

This can happen when C2Rust emits obsolete nightly feature gates. SafeMAP records that as a baseline failure and still evaluates the safe-first final output.

Glossary

Fully safe accepted unit
A generated Rust unit that passes strict safe acceptance and validation.
Migration plan
A JSON artifact describing the target Rust API, idioms, constraints, and validation rules.
C2Rust baseline
Reference translation used for comparison, usually unsafe and not SafeMAP final output.
Eligibility
The pre-translation classification that determines whether SafeMAP should attempt safe migration.
Differential testing
Comparing original C behavior and generated Rust behavior on the same inputs when practical.
